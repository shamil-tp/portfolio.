N-jin Web Crawler

Search Engine Crawler · 250,000+ Websites Indexed

Project Overview

N-jin Web Crawler is a powerful, production-grade web crawling engine built to power a custom search engine. Starting from a base URL, the crawler recursively traverses the web — extracting internal links, normalising URLs, and indexing page data at scale.

Built by a 3-person research team after extensive study of how real search engines work, this is not a toy crawler — it successfully indexed over 250,000 websites before being paused due to resource constraints. The system reached a point where speed and resource usage had to be carefully balanced, with the final version representing the optimal equilibrium between crawl velocity and compute efficiency.

The project's most distinctive feature is its documented evolution across 10 parser versions (v1–v10), each one a deliberate improvement over the last — making the codebase a living record of engineering iteration and research-driven development.

How The Crawler Works

The crawler operates as a recursive link-following engine — seeded with a base URL, it processes pages, extracts all valid internal links, normalises them to avoid duplicates, and recursively queues new pages for crawling.

  • Seed URL Input: The crawl starts from a given base URL, passed through the Express interface or directly via the CLI.
  • HTML Fetching: Each URL is fetched via HTTP. The raw HTML response is passed to the active parser version for processing.
  • Link Extraction: The parser scans the DOM for all anchor tags, extracts href attributes, and filters for internal links matching the base domain.
  • URL Normalisation: Duplicate URLs, trailing slashes, query string variants, and fragment identifiers are normalised to prevent re-crawling the same page.
  • Visit Tracking: Each URL's visit frequency is tracked — recording how many times a page was discovered across different crawl paths.
  • Recursive Queue Processing: Newly discovered URLs are pushed to the crawl queue. The system processes these iteratively, respecting concurrency limits tuned per version.
  • Data Persistence: Crawled page data and metadata are stored for use by the N-jin search engine's indexing layer.
  • Report Generation: A reporting module compiles crawl results — pages discovered, link counts, visit frequencies — into a structured report.

Parser Evolution: v1 → v10

The parsers/ folder tells the full engineering story — 10 versions of the crawler parser, each a deliberate research-driven improvement over the previous. One version hit peak crawl speed but was too resource-heavy and had to be rolled back. The final version finds the optimal balance between speed and efficiency.

v1
Foundation — Basic Recursive Crawler
Initial proof-of-concept. Single-threaded recursive crawl from a base URL. Basic link extraction using string parsing. No normalisation, no deduplication. Established the core crawl loop.
Baseline
v2
URL Normalisation & Deduplication
Introduced URL normalisation logic — handling trailing slashes, fragments, and query strings. Added a visited-URLs Set to prevent infinite re-crawling loops. Dramatically improved stability.
Stability +
v3
DOM Parser Integration
Replaced raw string-based parsing with a proper HTML DOM parser (jsdom/cheerio). Increased link extraction accuracy and robustness against malformed HTML.
Accuracy +
v4
Concurrency — Parallel Fetching
Introduced concurrent URL fetching using Promise.all batching. Multiple pages crawled simultaneously. First major speed improvement — crawl throughput increased significantly.
Speed ++
v5
Error Handling & Timeout Guards
Added per-request timeouts, HTTP error handling (4xx/5xx), and retry logic. Prevented the crawler from hanging on unresponsive servers. Critical for long crawl sessions.
Resilience +
v6
Visit Frequency Tracking
Added a Map-based visit counter — tracking how many times each page URL was discovered across different crawl paths. Foundation for PageRank-style relevance scoring in N-jin.
Indexing +
v7
Peak Speed — Aggressive Concurrency
Maximum parallelism — large batch sizes, minimal throttling, aggressive concurrent fetching. Fastest version built. Could crawl thousands of pages per minute. However, this version consumed extreme RAM and CPU — unsustainable for long sessions.
Fastest Ever Resource Critical ⚠
v8
Resource Throttling & Rate Limiting
Rolled back aggressive concurrency from v7. Introduced configurable concurrency limits, inter-request delays, and memory usage monitoring. Sacrificed some speed to regain stability for extended crawl sessions.
Balanced
v9
Queue Management & Depth Control
Replaced recursive call stack with an explicit queue data structure. Added crawl depth limiting to control how deep the crawler follows links. Prevented stack overflow on deep sites and enabled precise scope control.
Architecture +
v10
Final Production Version — Speed / Resource Balance
The culmination of all 9 iterations. Optimal concurrency tuned through real-world testing. Adaptive throttling based on response times. Full error resilience, depth control, deduplication, frequency tracking, and efficient memory usage. Successfully crawled 250,000+ websites.

Key Features

  • 250,000+ Pages Crawled: Real production-scale indexing — not a demo. Paused only due to infrastructure cost, not code limitations.
  • v1 → v10 Parser Evolution: Ten documented parser versions in the parsers/ folder — each a measurable improvement, showing a complete engineering research journey.
  • URL Normalisation Engine: Handles trailing slashes, query strings, fragments, and protocol variants — prevents duplicate crawling of the same logical page.
  • Visit Frequency Tracking: Maps how many times each URL is discovered — foundational data for relevance ranking in the N-jin search engine.
  • Adaptive Concurrency: Tunable parallel fetch batches — the final version finds the sweet spot between maximum speed and sustainable resource usage.
  • Explicit Queue Frontier: Queue-based (not recursive call-stack) crawl frontier — prevents stack overflow on deep sites, enables precise depth control.
  • Error Resilience: Per-request timeouts, HTTP error handling, and retry logic ensure the crawler survives unresponsive or broken servers during long sessions.
  • N-jin Search Interface: EJS-powered web UI for interacting with the search engine and triggering crawl operations.
  • Research-Backed Design: Three-person team conducted deep research into real search engine architecture before and during development — not built from tutorials.
